2. 설계 목적 — 역할별 최소 단위

이 워크플로우는 각 역할이 자기 단위만 보고 일할 수 있도록 최소 단위로 쪼갠 결과입니다. 모든 판단 근거가 지라 안에 있어 지라가 단일 소스(single source) 역할을 합니다. 신입이라면 "내가 어느 역할인지"부터 확인하세요 — 대부분의 작업자는 서브 이슈만 다룹니다.

🛠 빌드매니저

지라만 보고 일합니다. 메인 병합 · 릴리즈 구성 · QA 트리거 · 라이브 패치 판단을 이슈 상태/링크만으로 진행.

👤 작업자

서브 이슈 위주로만 작업. 자기 일감(commit 단위)을 대기→진행→완료로 처리하면 끝, 릴리즈 흐름은 신경 쓸 필요 없음.

🔍 QA

QA만 합니다. 릴리즈 단위로 들어온 빌드를 통합 검증하고, 실패분은 신규 이슈로 발행(링크)하면 끝.

왜 forward-only + 이슈 링크인가 — 일감을 뒤로 되돌리지 않고 항상 완료/취소로 닫은 뒤 후속을 신규 이슈로 분리·링크하면, 각 역할은 "열려 있는 자기 일감"만 보면 됩니다. 되돌아오는 일감이 없으니 상태가 단순해지고, 링크로 히스토리는 그대로 보존됩니다. (forward-only·이슈 링크의 구체적 동작은 4~6장에서 다룹니다.)

4. 이슈 타입별 워크플로우

서브 이슈 — 작업 단위 (commit)

대기 업무개시→ 진행 작업 완료→ 완료

진행하던 작업을 이번 버전에 넣지 않기로 하면 취소(미반영)로 빠집니다.

메인 이슈 — 병합 브랜치 단위

대기 업무개시→ 진행 모든 서브 일감 완료→ 병합 apk 테스트→ QA 통과→ 완료

하위 서브가 모두 종결(완료 또는 취소)되면 체크 후 병합 → 병합 상태. 병합된 메인들이 모여 릴리즈를 이루면 apk 테스트 → QA를 거쳐 완료됩니다.
※ 릴리즈 범주에서는 완료·취소 모두 "종결"로 보아 게이트를 통과시킵니다.

QA 결과와 무관하게 일감은 앞으로만 진행(forward-only) — 실패해도 뒤로 돌아가지 않습니다. QA 실패분도 완료로 종결하고, 후속 작업은 신규 이슈로 발행해 같은 버전 또는 다음 버전에 타겟팅합니다. 최악의 경우 종결과 함께 롤백.
※ 완료 = "이 일감의 수명이 끝남"이지 "검증 통과"가 아닙니다. 취소(미반영)는 애초에 이번 버전에 넣지 않은 경우로 의미가 다릅니다.
※ 종료·취소된 일감과 그 후속 신규 일감은 이슈 링크로 연결해 히스토리를 끊기지 않게 이어갑니다.
대기 진행 병합 QA 완료 취소(미반영)

5. 버전 = 지라 릴리즈 단위

병합된 메인들이 모여 하나의 릴리즈(버전)를 구성합니다. 각 릴리즈는 1번의 점검에 반영되는 패치 단위이며, 지라 릴리즈 탭에서 버전별로 관리됩니다.

진행률 막대: 초록=완료 · 파랑=진행 · 회색=잔여

통합 QA (forward-only) — 릴리즈 단위 병합이 끝나면 그 버전에서 통합 QA를 진행합니다. 실패하더라도 해당 일감은 그대로 완료 종결하고(뒤로 돌리지 않음), 후속은 신규 지라를 발행해 같은 버전 또는 다음 버전에 타겟팅합니다. 최악의 경우 종결과 함께 롤백합니다.
_live 접미 릴리즈 — 점검 전 QA로는 끝낼 수 없고 라이브에서만 확인해야 종료되는 이슈 (특정 타이밍 발생, 로그·모니터링 검증 등)를 모으는 별도 버전입니다. 해당 이슈가 없는 버전에는 생기지 않을 수도 있습니다(선택적).

6. 릴리즈 패치까지의 전체 흐름

  1. 서브 작업 진행 & 완료
    각 서브 일감(commit 단위)을 대기 → 진행 → 완료로 처리. 빠지는 작업은 취소(미반영).
  2. 메인 단위 병합
    한 메인의 모든 서브가 종결(완료 또는 취소)되면 체크 후 병합 → 메인이 병합 상태로.
  3. 릴리즈(버전) 구성
    병합된 메인들이 모여 하나의 릴리즈를 구성. 이번 점검에 반영될 단위입니다.
  4. apk 테스트 → 통합 QA
    릴리즈 단위로 apk 테스트 후 통합 QA 진행. 실패분도 완료로 종결하고 신규 이슈를 발행해 같은/다음 버전에 타겟팅(최악의 경우 롤백).
  5. 완료 & 라이브 패치
    QA 통과 시 메인은 완료 → 해당 릴리즈를 점검 시 라이브에 패치(예: 1.6.10 → 1.6.10_live).
참고 — 에픽은 1번의 패치로 끝나지 않는 작업을 묶는 상위 단위이므로, 위 1~5단계(릴리즈 단위 사이클)가 여러 번 반복되며 그 진행 상황을 추적합니다.

7. 실전 운영 팁

① 히스토리 코멘트는 메인 일감에 남긴다
서브에서 진행한 브랜치명 등은 메인 일감 코멘트로 모읍니다. 그래야 빌드매니저가 메인 하나만 봐도 문맥이 서고, ver 넣기 전 작업물을 모아 유니티 에디터로 목표 달성을 확인한 뒤 병합을 넣을 수 있습니다. (서브 = 작업 진행 / 메인 = 히스토리·검증·병합 판단의 기준점)
② QA 실패 = 항상 신규 이슈는 아니다 (강제하지 말 것)
실패분은 완료로 종결하되, 후속 이슈 발행은 케이스마다 판단합니다. 바로 고칠 수도, 일정상 다음 버전으로 미룰 수도 있기 때문에 "무조건 이슈 생성"으로 강제하면 오히려 부정확해집니다. 대신 후속 이슈가 생길 때만 이슈 링크로 히스토리를 잇습니다.
③ (참고) Smart Commit — 미래 자동화 옵션
커밋 메시지에 남기는 지라 URL은 일감 바로가기 하이퍼링크용이지만, 그 URL 안에는 이슈 키(예: AS-123)가 들어 있습니다. repo↔지라 연동을 걸고 키 파싱만 정리하면, 커밋이 해당 일감에 자동 링크되고 #comment 명령으로 커밋 내용을 코멘트로 남길 수도 있습니다. ※ 단, 자동 코멘트는 커밋 author 이메일 = 지라 계정 이메일 매칭이 필요하고, 연동은 지라 인스턴스마다 1회 설정해야 합니다. (현 단계에선 미적용)
④ 서브 → 메인 상태 자동 전환 — 우선 "진행"만 적용
지라 자동화 규칙(Automation)으로 서브 이슈 중 1개라도 "진행"으로 바뀌면 메인 이슈를 "진행"으로 전환하는 것만 우선 겁니다. ※ 서브는 중간에 추가될 수 있어 "모든 서브 종결 → 메인 병합" 같은 자동화는 깨지기 쉬우므로 적용하지 않습니다. "하나라도 진행 → 메인 진행"은 그런 영향 없이 안전합니다. (병합·QA 등 사람 게이트가 있는 전환은 수동 유지)

8. 용어 정리 (부록)

에픽 이슈1번의 패치로 완료되지 않는, 여러 버전에 걸친 작업 묶음 단위.
메인 이슈하나의 버전에 병합할 브랜치 단위. 서브들의 묶음.
서브 이슈실제 작업 단위(= commit). 가장 작은 일감.
릴리즈(버전)병합된 메인들의 집합. 1번의 점검에 반영되는 패치 단위(지라 릴리즈로 관리).
통합 QA릴리즈 전체를 묶어 진행하는 QA. 결과에 따라 롤백·추가 이슈 반영.
점검릴리즈를 라이브에 반영(패치)하는 시점.
완료일감의 수명이 끝난 종결 상태. "검증 통과"가 아님 — QA 실패분도 완료로 닫고 후속은 신규 이슈로 분리.
취소(미반영)애초에 이번 버전에 넣지 않기로 한 서브/메인의 종결 분기. 완료와 의미가 다름.
종결릴리즈 범주에서 완료 또는 취소 상태. 상위 게이트(병합·에픽 완료)는 종결 기준으로 판단.
이슈 링크종료·취소된 일감과 후속 신규 일감을 연결해 히스토리를 잇는 장치.
